# Home-IDS-NSM-Security-Onion

In this lab, I deployed Security Onion as an IDS/NSM receiving traffic from a lightweight VM running Alpine Linux and a VM acting as an attacker running Kali Linux. I connected the machines to a SPAN port and sent the traffic to Security Onion for analysis. The SPAN port was configured with Open vSwitch.
I also created a custom Suricata rule to test its functionality.

## Network configuration with Open vSwitch

The machines were deployed using QEMU/KVM+libvirt, which facilitated the network configuration of the VMs.

First, we must download Open vSwitch. Since I am using Fedora, this is simple. Just run:
`sudo dnf install openvswitch`

For more information visit: https://docs.openvswitch.org/en/latest/intro/install/#installation-from-packages

Use `ovs-vsctl --version` to confirm correct installation.

After this we must enable Open vSwitch with: `sudo systemctl enable --now openvswitch`

Once Open vSwitch is enabled, we can create the port that we will use to create the mirror. In this exercise, we want the VMs to be isolated from the internet, so we will remove any default IP generated by the port.
Also, before setting up the SPAN port, we must configure the VMs and do so with the VMs turned on, so now we will just create the port and configure it later.
We execute the following commands:

Create the port: `sudo ovs-vsctl add-br [Name-of-the-port]`
Check if it's work: `sudo ovs-vsctl show`
Open the port: `sudo ip link set [Name-of-the-port] up`
Eliminate IP:`sudo ip addr flush dev [Name-of-the-port]`

## Deploy SPAN port

While keeping the VMs powered on, we must deploy the SPAN port.

To do this, we must first verify the name of each VM. Using libvirt (QEMU/KVM), this can be seen in the network configuration tab. The name is usually something similar to “vnetX.”

Now we run this in the terminal:
`udo ovs-vsctl \
  -- --id=@p get port vnetX \ #Change the X for the number of Security Onion port
  -- --id=@m create mirror name=span-mirror select-all=true output-port=@p \
  -- set bridge [Name-of-the-port] mirrors=@m`

We can use this from check if it works:`sudo ovs-vsctl list mirror`
Also, if you want to delete a port, you can use `sudo ovs-vsctl clear bridge br-span mirrors`, and if you want to use specific ports instead of `select-all=true`, use `select-src-port` and `select-dst-port` to select the destination ports.
Another detail: if it does not allow you to use the port name, you can use its UUID. To verify it, use `sudo ovs-vsctl list Port`.

## Configuration of services

Now you have deployed the SPAN port, but there are still a few things to configure.

#### IP: 
You need to assign an IP address to both the attacker VM (Kali) and the victim VM (Alpine). To do this, run the following command:
`ip addr add X.X.X.X/24 dev eth1` Choose the IP address you want.

Now you can execute a ping on the machine and capture the traffic with tcpdump in Security Onion manager to be sure that the SPAN port works.

#### Suricate rules: 
This is not a specific projects about Suricata rules. I'm just going to give a little background on how it works. For more information, please visit:https://docs.suricata.io/en/latest/rules/

To provide some context, the rule format consists of three parts:
First, **the action** determines what happens when the rule is met. 
Second, **the header** defines the protocol, IP, ports, and address.
Third, **the rule options** define the specifications of the rule.

This is a simple rule I created for HTTP connections on non-standard ports, following MITRE T1071.001.
`alert http $HOME_NET nay -> $EXTERNAL_NET !80,!443!,!8080(msg:"[LAB] HTTP in weird port - possible C2 or simulated beacon"; flow:to_server; established; classtype: policy-violation; sid:1000001; rev1;)`
